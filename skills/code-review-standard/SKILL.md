# 代码审阅标准

## 审阅维度

从以下 5 个维度审阅代码，每个维度 1-10 分：

### 1. 代码质量 (quality)
- **可读性**：变量命名清晰、逻辑直观、注释适当
- **可维护性**：模块化、单一职责、低耦合
- **代码风格**：符合语言规范、一致的格式

### 2. 功能正确性 (correctness)
- **逻辑正确性**：算法正确、边界条件处理
- **错误处理**：异常捕获、错误恢复
- **输入验证**：参数检查、类型验证

### 3. 性能 (performance)
- **时间复杂度**：算法效率
- **空间复杂度**：内存使用
- **资源使用**：避免不必要的计算

### 4. 安全性 (security)
- **输入验证**：防止注入攻击
- **权限控制**：最小权限原则
- **数据保护**：敏感信息处理

### 5. 测试 (testing)
- **测试覆盖**：关键路径覆盖
- **测试质量**：断言有效性
- **边界测试**：边界条件测试

## 评分规则

### 总分计算
```
overall_score = (quality * 0.2) + (correctness * 0.3) + (performance * 0.2) + (security * 0.15) + (testing * 0.15)
```

### 严重程度定义

| 严重程度 | 定义 | 处理方式 |
|----------|------|----------|
| **high** | 影响功能正确性或安全性 | 必须修复 |
| **medium** | 影响性能或可维护性 | 建议修复 |
| **low** | 代码风格或最佳实践 | 可选修复 |

### 达标标准

| 条件 | 说明 |
|------|------|
| 总分 >= 目标分数 | 默认 9.0 分 |
| 无 high 严重度问题 | 必须满足 |
| medium 问题 <= 2 个 | 建议满足 |

## 输出格式

审阅结果必须返回以下 JSON 格式：

```json
{
  "scores": {
    "quality": 8,
    "correctness": 7,
    "performance": 9,
    "security": 6,
    "testing": 5
  },
  "overall_score": 7.0,
  "issues": [
    {
      "severity": "high",
      "category": "performance",
      "line": 42,
      "description": "递归实现效率低下",
      "suggestion": "建议使用动态规划"
    }
  ],
  "strengths": ["代码简洁", "命名清晰"],
  "summary": "总体评价"
}
```

## 修复要求

根据审阅问题修复代码时：

1. **优先级**：high > medium > low
2. **保持功能**：修复不能改变原有功能
3. **最小修改**：只修改必要的部分
4. **添加注释**：复杂逻辑添加注释
5. **添加测试**：关键路径添加测试用例
