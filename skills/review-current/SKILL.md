# 审阅当前改动

审阅当前会话中最近修改的代码文件，提供即时反馈。

## 触发条件

当用户的请求包含以下意图时使用此技能：
- "审阅当前改动"、"审阅刚才的修改"
- "review changes"、"review current"
- "检查刚才写的代码"、"看看刚才的修改"
- "刚才改的代码有问题吗"

## 执行步骤

### 步骤 1：识别最近修改的文件

查找当前会话中最近修改的代码文件：

```bash
# 查找最近 30 分钟内修改的代码文件
find . -name "*.py" -o -name "*.js" -o -name "*.ts" -o -name "*.java" -o -name "*.go" -o -name "*.rs" | xargs ls -lt 2>/dev/null | head -10
```

或者从会话上下文中识别：用户要求写入/编辑的文件。

### 步骤 2：读取文件内容

读取每个修改过的文件的完整内容。

### 步骤 3：调用 code-reviewer 审阅

对每个文件调用 `mcp__code-reviewer__review_code` 工具：

```
调用 mcp__code-reviewer__review_code 审阅以下代码：
文件：{file_path}
代码：{file_content}
```

### 步骤 4：展示结果

用以下格式展示审阅结果：

```
📝 审阅了 X 个文件

### file1.py — 7.5/10
⚠️ 2 个问题：
- [medium] 性能：...
- [low] 测试：...

### file2.py — 8.8/10
✅ 无中高严重度问题

需要我自动修复这些问题吗？
```

### 步骤 5：询问是否自动修复

如果发现中高严重度问题，询问用户是否需要自动修复。如果用户同意：
1. 根据审阅意见修改代码
2. 写回文件
3. 再次审阅确认

## 与 auto-review-loop 的区别

| 特性 | review-current | auto-review-loop |
|------|---------------|-----------------|
| 触发方式 | 手动（用户说"审阅当前改动"） | 手动（用户说"循环审阅"） |
| 循环次数 | 单次审阅 + 可选修复 | 自动循环到达标 |
| 适用场景 | 写代码过程中即时检查 | 代码完成后深度优化 |
| 速度 | 快（单次审阅） | 慢（多轮循环） |
